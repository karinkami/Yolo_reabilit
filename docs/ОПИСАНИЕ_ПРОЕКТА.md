# Описание проекта YOLO Reabilit
---

## 1. Назначение и идея

**YOLO Reabilit** — веб-система реабилитации: пациент выполняет назначенные врачом упражнения перед веб-камерой, получает обратную связь в реальном времени; врач видит историю и статистику.

**Ключевая идея архитектуры:** **гибридная система** — модуль оценки позы по видео даёт координаты суставов (для текущей реализации этого достаточно), модуль на правилах — **встраиваемая экспертная система**: пороги углов, фазы, подсчёт повторов и тексты обратной связи.

---

## 2. Реализация

| Область | Реализация |
|---------|------------|
| **Оценка позы** | Модуль pose estimation: 17 ключевых точек тела на кадре (достаточно для текущей задачи) |
| **Реальное время** | MJPEG-поток + опрос состояния ~280 ms, подсказки без перезагрузки страницы |
| **Клинический поток** | Врач → назначение (упражнение, сторона, повторы, связка 2/2) → пациент → отчёт |
| **Стабильность трекинга** | Один человек в кадре, IoU-бокс, приоритет рабочей руки/ноги, сглаживание координат |
| **Связки сторон** | Левая + правая (или две ноги при приседе) в одном пункте плана; прогресс 1/2 в истории |
| **Завершение назначения** | При достижении цели по повторам запись уходит в статистику, назначение снимается с активного плана |
| **Статистика** | Графики, средняя оценка, история с датами в локальном времени пользователя |
| **Интерфейс** | React SPA: кабинет врача, тренировка, статистика пациента |
| **Экспертная система** | База знаний в коде упражнений + механизм вывода `detect()` на каждом кадре |

## 4. Роли и сценарии

### Врач

- Список пациентов, карточка (диагноз, рекомендации в `notes`).
- Назначение упражнения: ключ, **сторона** (left/right), **цель повторов**, опционально **связка** на обе руки (`assignment_group_id`).
- Просмотр статистики и последней тренировки пациента.

### Пациент

- План активных назначений (после цели упражнение убирается из списка).
- Страница тренировки `/patient/rehab/`: камера, счётчик «N / цель», угол, фаза, подсказки, персональные рекомендации врача (текст, без влияния на пороги).
- Статистика и таблица «Детализация по упражнениям».

Персональные рекомендации врача (`PatientProfile.notes`) отображаются на тренировке как контекст; алгоритм углов их **не читает** — это осознанное разделение «человек + автоматика».

## 5. Модуль компьютерного зрения (оценка позы)

### 5.1. Задача и почему этого достаточно

Для **текущей реализации** проекта нужно одно: стабильно получать координаты суставов в кадре веб-камеры. Эту задачу решает **pose estimation** (оценка позы человека).

В проекте подключён готовый модуль **оценки позы** (YOLO11 pose, Ultralytics) — для **текущей** постановки задачи этого **достаточно**:

- не требуется собирать свой датасет и обучать сеть под каждое упражнение;
- на обычном ПК с веб-камерой уже можно считать углы и повторы в реальном времени;
- основная «интеллектуальная» часть реабилитации вынесена в **экспертную систему на правилах** — туда входят пороги, фазы и тексты для врача и пациента.

Такой разделение осознанное: **компьютерное зрение даёт числа**, **экспертная система принимает решения** по протоколу ЛФК. Для пилота и дипломной версии этого **достаточно**; отдельное обучение модели под каждого пациента в scope проекта не закладывалось.

| Этап | Что делает модуль | Результат |
|------|-------------------|-----------|
| Детекция человека | Находит фигуру в кадре | Прямоугольник (бокс) вокруг человека |
| Ключевые точки | 17 точек формата COCO | `left_shoulder`, `right_elbow`, `left_hip`, … |
| Передача дальше | Координаты в пикселях + уверенность | Вход для `geometry.py` и `detect()` |

По умолчанию используется компактная конфигурация **`yolo11n-pose`** (быстрее на CPU). При необходимости точнее — переменная окружения `YOLO_POSE_MODEL` (например `yolo11s-pose.pt`). Это **усиление качества**, а не обязательное условие работы системы.

Упражнение («отведение», «присед» и т.д.) задаёт **врач в назначении**; отдельной модели «на скапцию» или «на присед» в проекте нет и **не требуется** для выбранной архитектуры.

### 5.2. Пайплайн на одном кадре

```text
Кадр с веб-камеры (BGR)
    → зеркало по горизонтали (как у веб-камеры)
    → YOLO.predict (размер imgsz, stride для экономии CPU)
    → N человек в кадре?
         → оценка каждого кандидата (_person_score_for_index)
         → PersonBoxStabilizer: тот же человек по IoU бокса
    → KeypointSmoother: экспоненциальное сглаживание x,y
    → extract keypoints для active_side
    → exercise.detect(keypoints, side, target_reps)
    → update_state → JSON feedback
```

Файлы: `app/camera.py`, `app/pose_tracking.py`, `app/pose_overlay.py`.

### 5.3. Выбор одного человека в кадре

Если в кадре несколько людей, для каждой детекции считается **взвешенная оценка** (простая формула, не нейросеть):

```text
S = 0.58 * C_arm + 0.26 * C_center + 0.16 * min(A_norm, 0.85)
```

где:

- `C_arm` — средняя уверенность модели на точках **рабочей** руки (плечо, локоть, кисть, бедро той же стороны) или на обеих руках для дыхания;
- `C_center` — насколько центр человека близок к центру кадра;
- `A_norm` — нормированная площадь бокса детекции.

**Стабилизатор:** если бокс пересекается с прошлым кадром (IoU не ниже порога), остаёмся на том же человеке; иначе переключаемся только если новый кандидат заметно лучше по `S`.

### 5.4. Сглаживание координат

Для каждой точки применяется **экспоненциальное сглаживание** (EMA):

```text
p_t = alpha * p_raw + (1 - alpha) * p_{t-1}
```

- `alpha` меньше для фоновых точек (`POSE_SMOOTH_ALPHA`, около 0.52) — плавнее картинка;
- `alpha` больше для рабочей руки/ноги (`POSE_SMOOTH_ALPHA_ACTIVE`, около 0.82) — меньше задержка на активной конечности.

Это снижает дрожание угла и ложные срабатывания фаз.

### 5.5. Зеркало и сторона

Кадр отражается по горизонтали перед инференсом (`cv2.flip`).

| Тип упражнения | Сторона в модели |
|----------------|------------------|
| Рука (профиль) | После зеркала: `get_model_side(user_side)` — рабочая рука в кадре |
| Нога, присед | **Без** инверсии: `get_tracking_side` = сторона из назначения врача |
| Подъём вперёд | Как плечевые — **боком** к камере |

Пациент в профиле стоит так, что **рабочая** конечность ближе к объективу.

### 5.6. Присед и чувствительные упражнения

Для `partial_squat`, `forward_raise`, `arm_raise` pose-инференс выполняется **каждый кадр** (не через stride), чтобы не терять повторы.

**Присед:**

- при выборе человека в кадре — бонус за профиль (`squat_profile.profile_score_from_detection`);
- точки ноги — сторона с **лучшей видимостью** бедро–колено–щиколотка (`extract_leg_points_any_visible`);
- угол сглаживается EMA; повтор — заметное сгибание от стойки и возврат почти в стойку (`app/exercises/squat_thresholds.py`, `partial_squat.py`).

## 6. Математика углов и метрик

Для большинства упражнений угол в суставе считается **через скалярное произведение двух векторов** в плоскости кадра (2D). Это реализовано в коде, а не только в теории.

### 6.1. Скалярное произведение и угол в суставе

Вершина угла — точка **B** (например, плечо или колено), концы — **A** и **C** (бедро и локоть, колено и голеностоп и т.д.):

```text
        A
         \
          \
           B ———— C
```

Координаты точек в пикселях: A = (ax, ay), B = (bx, by), C = (cx, cy).

**Шаг 1 — векторы от вершины B:**

```text
BA = (ax - bx, ay - by)
BC = (cx - bx, cy - by)
```

**Шаг 2 — скалярное произведение:**

```text
dot = BA · BC = (ax - bx)*(cx - bx) + (ay - by)*(cy - by)
```

В коде (`app/exercises/geometry.py`):

```python
ba = (ax - bx, ay - by)
bc = (cx - bx, cy - by)
dot = ba[0] * bc[0] + ba[1] * bc[1]
```

**Шаг 3 — длины векторов:**

```text
|BA| = sqrt((ax - bx)^2 + (ay - by)^2)
|BC| = sqrt((cx - bx)^2 + (cy - by)^2)
```

**Шаг 4 — косинус угла:**

```text
cos_theta = dot / (|BA| * |BC|)
```

Если длина нулевая (точка не видна), угол = 0°.

**Шаг 5 — угол в градусах:**

```text
theta = degrees( acos( clip(cos_theta, -1, 1) ) )
```

`clip` ограничивает cos_theta диапазоном [-1, 1] из‑за погрешности вычислений.

Функция: **`angle_three_points(a, b, c)`** в `app/exercises/geometry.py` — одна на все упражнения с угловой метрикой.

#### Где вызывается `angle_three_points`

| Упражнение | Точки A — B — C | Сустав |
|------------|-----------------|--------|
| Отведение, скапция, подъём вперёд/вверх | бедро — **плечо** — локоть | плечо |
| Сгибание в локте | плечо — **локоть** — кисть | локоть |
| Присед, разгибание колена | бедро — **колено** — голеностоп | колено |

Чем выше рука в профиле в 2D, тем обычно больше угол theta для цепочки бедро–плечо–локоть.  
Выпрямленная нога даёт большой угол (примерно 150–180°); при сгибании в колене угол **уменьшается**.

**Исключение:** `breathing_arms` / `breathing_arms_slow` для двух рук в кадре **не** используют скалярное произведение для повтора — там метрика `lift_norm` (раздел 6.3).

### 6.2. Скорость изменения угла (для подсказок)

```text
скорость_угла (град/с) ≈ (theta_t - theta_{t-1}) / (t_t - t_{t-1})
```

`angle_velocity_deg_s` в `BaseExercise` — если скорость слишком высокая, подсказка «медленнее», «без рывка» (`live_coaching.py`).

### 6.3. Дыхание с двумя руками — нормированный подъём кисти

Когда в кадре обе руки, повтор считается по **высоте кисти** относительно плеча, нормированной на ширину плеч:

```text
span = max( 55, расстояние между левым и правым плечом )
lift_norm = (y_плеча - y_кисти) / span
```

(ось Y на изображении направлена вниз — при подъёме кисти `lift_norm` растёт.)

Пороги: `LIFT_DOWN` ≈ 0.12, `LIFT_UP` ≈ 0.40.

Симметрия рук:

```text
asymmetry = | lift_norm_левая - lift_norm_правая |
```

Повтор засчитывается при подъёме обеих рук и малой асимметрии.

Для отображения на экране «угол» масштабируется линейно в диапазон 118°–150°:

```text
угол_на_экране = 150 - lift_norm * (150 - 118)
```

### 6.4. IoU боксов (пересечение детекций)

```text
IoU = площадь_пересечения / площадь_объединения
```

Используется, чтобы не «перепрыгивать» на другого человека между кадрами (`pose_tracking.py`).

## 7. Экспертная система (модуль правил)

Блок после YOLO по архитектуре соответствует **встраиваемой экспертной системе** (rule-based expert system): знания реабилитации закодированы как **если–то** правила над числовыми признаками (углы, фазы, время), без отдельного ML для решения «засчитать повтор или нет».


### 7.1. Соответствие классической архитектуре ЭС

| Компонент экспертной системы | Реализация в YOLO Reabilit | Где в проекте |
|------------------------------|----------------------------|---------------|
| **База знаний (Knowledge Base)** | Пороги углов, границы фаз, интервалы между повторами, условия блокировки повтора, шаблоны подсказок | `app/exercises/*.py`, `live_coaching.py`, `exercise_guides_defaults.py`, опционально `exercise_guides` в БД |
| **Рабочая память (Working Memory)** | Текущий угол, фаза, число повторов, флаги `ready`, время последнего повтора, скорость угла | объект `BaseExercise` + `app/state.py` на пользователя |
| **Механизм вывода (Inference Engine)** | На каждом кадре: вычислить признаки → проверить условия → обновить фазу/счётчик → сформировать вывод | метод `detect()` в классе упражнения |
| **Интерфейс пользователя** | Тексты `feedback`, `correctness`, `tips` на странице тренировки | React + `/api/feedback` |
| **Источник знаний (эксперт)** | Правила отражают типичные критерии ЛФК: амплитуда, темп, исходное положение, симметрия для дыхания | заложено при разработке порогов; врач задаёт **тип** упражнения и **цель** через назначение |

### 7.2. Пример правила в логике (если–то)

Для отведения плеча (упрощённо, на русском):

```text
ЕСЛИ  ready = истина
 И  phase = "down"
 И  угол(бедро, плечо, локоть) ≥ top_threshold
 И  прошло ≥ min_rep_interval с прошлого повтора
 И  угол < too_high_block
ТО  phase := "up"
    reps := reps + 1
    feedback := "Верх отведения. Повтор N из цели."
```

Таких условий в одном упражнении — десятки ветвей (`if/elif` в Python). Вместе они образуют **детерминированный автомат** фаз движения.

### 7.3. Что входит в базу знаний

1. **Числовые пороги** — например `top_threshold = 68°` для отведения в профиле.  
2. **Правила фаз** — `waiting_start` → `down` → `up` → снова `down`.  
3. **Правила качества** — слишком быстрый подъём (`angle_velocity_deg_s`), асимметрия рук при дыхании.  
4. **Текстовые выводы** — что показать пациенту при срабатывании правила.  
5. **Справочные тексты** — «как выполнять», «типичные ошибки» (гайды; на логику повторов не влияют).

Знания **не** вынесены в отдельный редактируемый файл правил (как в классических ЭС 80–90-х) — они в модулях Python. Для дипломного проекта это нормально: база знаний **явная и читаемая**, но связана с кодом.

### 7.4. Разделение: оценка позы и эксперт

```text
┌──────────────────┐     координаты точек      ┌─────────────────────────┐
│  Модуль позы     │ ────────────────────────► │  Экспертная система   │
│  (видео → точки) │     угол, lift_norm, …    │  rules + detect()     │
└──────────────────┘                           └───────────┬─────────────┘
                                                             │
                                                             ▼
                                                   feedback, reps, phase
```

Модуль зрения выдаёт координаты суставов; критерии «повтор засчитан / не засчитан» и тексты для пациента — зона **экспертной системы**. Для текущего объёма проекта такое разделение **достаточно и обосновано**.

## 8. Логика упражнений (реализация базы знаний)

Каждое упражнение — класс в `app/exercises/<key>.py`, наследник `BaseExercise` (конкретная «подсистема» эксперта под тип движения).

### 8.1. Фазы и повторы

Типичный автомат для одной руки:

```text
waiting_start → (рука внизу) → ready, phase=down
down → (угол ≥ top_threshold) → phase=up, reps += 1  [если can_count_rep]
up  → (угол ≤ return_down) → phase=down
```

**`can_count_rep`:** между повторами минимум `min_rep_interval` секунд (защита от двойного счёта на одном подъёме).

**Пример порогов отведения** (`shoulder_abduction.py`, профиль):

| Параметр | Значение | Смысл |
|----------|----------|--------|
| start_threshold | 30° | Исход — рука внизу |
| top_threshold | 68° | Верх — повтор засчитан |
| return_down_threshold | 38° | Вернул вниз — готов к следующему |
| too_high_block | 84° | Слишком высоко — не засчитывать |

**Частичный присед** — относительно угла **стойки** в кадре (не фиксированные 90°/120°):

```text
drop = stand_angle − angle(бедро, колено, щиколотка)
присед подтверждён: drop ≥ BENT_DROP (~14°) несколько кадров
повтор: был присед и drop ≤ RETURN_DROP (~7°) к стойке
```

Пороги: `app/exercises/squat_thresholds.py`. Подъём вперёд и подъём вверх — рост угла от исхода «рука внизу», засчёт часто на опускании.

### 8.2. Каталог упражнений

| Ключ | Название | Основная метрика |
|------|----------|------------------|
| shoulder_abduction | Отведение в сторону | hip–shoulder–elbow |
| forward_raise | Подъём вперёд (боком к камере) | hip–shoulder–elbow, цикл вперёд-вверх → вниз |
| scaption_raise | Скапция | hip–shoulder–elbow |
| arm_raise | Подъём вверх | hip–shoulder–elbow |
| breathing_arms | Дыхание, обе руки | lift_norm / симметрия |
| elbow_flexion | Сгибание локтя | shoulder–elbow–wrist |
| partial_squat | Частичный присед | hip–knee–ankle; **стоя боком** к камере; в связке правая, затем левая нога |

Скапция и отведение в **2D-профиле** дают близкий рост одного угла; различие задаётся **назначением врача** и текстами/порогами

`app/exercises/live_coaching.py` — дополнительные правила в базе знаний поверх угла и скорости его изменения: «не поднимайте плечо», «ровнее темп», подсказки для дыхания и приседа.

## 9. Оценка результата и статистика

### 9.1. Процент при сохранении сессии

```text
score_percent = round( 100 * reps_completed / target_reps )
```

Результат ограничивается диапазоном 0–100%.

### 9.2. Связка «левая + правая»

| Ситуация | Оценка в истории | Текст |
|----------|------------------|--------|
| Только 1-я сторона | 50% | Цель не достигнута (1/2) |
| Обе стороны, цель выполнена | 100% | Цель достигнута (2/2) |
| Обе стороны, не все повторы | пропорционально | Цель не достигнута (2/2) |

Неполная связка **не входит** в среднюю оценку в статистике (чтобы не смешивать 50% и 100% в одно «75%»).

### 9.3. История

- Записи в `training_sessions` с UTC-временем.
- Свёртка связок: одна строка на попытку с максимальным `bundle_part` (`session_aggregate.py`).
- Графики: число подходов и средняя оценка по **локальным** календарным дням.

### 9.4. Снятие с плана после тренировки

При сохранении подхода (`persist_training_result`, API `/patient/rehab/api/complete`):

- если выполнено **≥ цели** по повторам — назначение помечается неактивным;
- для **связки** (две руки / два подхода приседа) снимаются **все** строки группы после одного успешного подхода с полной целью;
- при досрочной остановке без цели назначение остаётся в плане;
- при открытии тренировки вызывается `sync_completed_assignments` — подтягивает уже выполненные связки по истории.

Файлы: `app/assignment_completion.py`, `app/training_session_save.py`, `frontend/src/RehabTraining.jsx`.
